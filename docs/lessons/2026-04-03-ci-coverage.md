# CI カバレッジ計算のトラブルシューティング知見

日付: 2026-04-03

---

## 1. `go tool cover -func` の出力フォーマットと grep パターンの落とし穴

### 問題

CIの authorizer カバレッジチェックで `grep -v ":main\b"` を使って `main` 関数を除外しようとしていたが、
実際には除外できておらず `main`（0.0%）が平均計算に含まれ、74.8% が報告されていた。

### 原因

`go tool cover -func` の出力フォーマットはタブ区切り：

```
github.com/.../main.go:200:	main	0.0%
```

`:main` というパターンは `main.go:200:` の部分に登場しない（コロンの後はタブ）。
そのため `grep -v ":main\b"` はマッチせず、main 関数が除外されなかった。

**誤った平均計算（main 含む）：**
```
(87.9 + 100.0 + 100.0 + 86.1 + 0.0) / 5 = 74.8%
```

**正しい計算（main 除外）：**
```
(87.9 + 100.0 + 100.0 + 86.1) / 4 = 93.5%
```

### 解決策

`grep -v` の代わりに awk の条件式で第2フィールド（関数名）を直接チェックする：

```bash
# NG: タブ区切りのため :main にマッチしない
| grep -v ":main\b" \
| awk '{gsub(/%/, "", $3); sum += $3; count++} ...'

# OK: awk の $2（関数名フィールド）で直接判定
| awk '$2 != "main" {gsub(/%/, "", $3); sum += $3; count++} ...'
```

`go tool cover -func` の出力フィールド構成：
| フィールド | 内容 | 例 |
|-----------|------|----|
| `$1` | ファイルパス:行番号: | `main.go:200:` |
| `$2` | 関数名 | `main` |
| `$3` | カバレッジ | `0.0%` |

### 今後の対策

- **`go tool cover -func` の出力をパースする際は awk の `$2`（関数名）で判定する**
- `grep` でパターンマッチするより awk でフィールド指定の方が堅牢
- CIのカバレッジチェックを変更した際は、ローカルで `go test -coverprofile=coverage.out` を実行し
  `go tool cover -func=coverage.out | grep <package>` で実際の出力形式を確認する
